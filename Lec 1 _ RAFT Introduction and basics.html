Basic Terminologies

Consensus??

Consensus is how a collection of machines

work together as a coherent group

Used to guarantee overall system-reliability in

the presence of faulty nodes

Generally achieved by having nodes

coordinating with each other

In laymans terms, Consensus is a general

agreement on a value

In our project we are using distributed

replicated log that uses consensus to ensure that all state-machines have the same log

Image Source :https://medium.com/@isuruboyagane.16/what-is-consensus-in-distributed-system-6d51d0802b8c

But what are replicated logs and state machines?

State machine is a program that

takes in inputs and outputs a result

A log is a collection of operations

(client requests) that are to be executed on the state machines

Consensus ensures that the logs

are replicated across all nodes

As long as all the logs are identical,

the state machine will produce the same sequence of results

As long as majority of servers are

up, the consensus mechanism can ensure log replication

Server States

At any given time each server is in one of three states:

Follower:

Passive states Issue no requests Responds to req from leader

Leader :

Handles all client requests. If client contacts a follower, follower

redirects it to leader

Typically operate until they fail

Candidate :

Used to elect new leader Follower becomes a candidate after a

timeout

Image Credits : https://raft.github.io/raft.pdf

Terms

Time is divided into terms of arbitrary length Numbered with consecutive integers Each term begins with an election

A follower becomes a candidate and attempts to be

the leader

If candidate wins the election it becomes the leader Each server stores current term number which increases

over time

If a candidate or leader discovers that its term is out of

date it reverts back to being a follower

If a follower receives a request from leader with a stale

term, it rejects the request

Image Credits : https://raft.github.io/raft.pdf

RPC: AppendEntries RPC and RequestVote RPC

RAFT servers communicate using Remote Procedure

Calls (RPCs)

RequestVote RPC:

Initiated by candidates during election sent to other

servers

On the receiver side, the server can choose to

either accept or reject a RequestVote RPC

AppendEntries RPC:

Initiated by leaders to replicate log entries on

followers

Also provides a form of heartbeat to let the followers

know that the leader is still alive

Heartbeats and Election Timeouts

RAFT uses heartbeat mechanisms to trigger leader elections A server remains in follower state as long as it receives valid RPC from leader (or candidate)

Leader sends periodic heartbeats (T) in the form of empty AppendEntries RPC

i. I.e AppendEntry RPC with no log entry

If follower receives no communication over a period of time ( [T, 2T] - ), it assumes that leader has

failed

The follower becomes candidate and sends out RequestVote RPC to other servers A candidate continues to be in this state until :

  1. It wins the election
  2. Another server establishes itself a leader (or)
  3. A period of time goes by with no winner

Log structure

Log Entry = Log Index + Term + Command

The command is the clients requests to the

server

Term is the term number when the req was

received (Used to detect inconsistencies between logs)

Index identifies the position of entry

Under normal operation, each client req gets

appended to the Leaders log and the req is sent to followers using the heartbeat AppendEntry RPC

A log entry is committed once the leader that created

the entry replicated the entry on a majority of servers

Image Credits : https://raft.github.io/raft.pdf

Log structure (contd)

A committed entry in the log is an entry that the leader

has decided to be safe

All committed entries will eventually be executed

by the state machine.

The leader keeps track of highest index it knows to be

committed and includes that information in all heartbeats so other servers eventually find out.

Once a follower knows that a log entry is committed, it

applies the entry to its local state machine

Logs are stored on stable storage (Survives

crashes). The logs on each server node needs to be persisted
